# Technical Terms Introduced Before Use

When a term first appears, it should be explained in plain language. After introduction, use the term consistently without re-explaining.

## Criteria

| Score | Level | Description |
|-------|-------|-------------|
| 1 | Poor | Technical terms appear throughout assignments and check script messages without prior introduction. Users encounter jargon before they have any context for it. |
| 2 | Below Standard | Most terms are introduced, but several appear in assignment steps or check script failure messages before the content explains them. Users must infer meaning in multiple places. |
| 3 | Adequate | Terms are generally introduced before use in assignments. One or two terms appear in check script messages or assignment steps slightly before the content introduces them. |
| 4 | Good | All terms are introduced in plain language before they appear in assignment steps or check scripts. After introduction, terms are used without re-explanation. One minor gap may exist. |
| 5 | Excellent | Every technical term is introduced with a clear, plain-language explanation before it appears anywhere -- assignments, assignment steps, check script messages. Check script feedback reinforces vocabulary by using the exact introduced terms. |

Example of a good introduction:

```markdown
Before you can commit changes, you need to **stage** them. The staging area
(also called the "index") is like a preparation zone for your next commit.
```

After introduction, use the term consistently without re-explaining. Check script feedback messages should use the introduced terms, reinforcing vocabulary.

## What to Watch For

- Technical terms used in check script messages that haven't been taught yet
