# Custom VM Image (Packer)

Pre-built GCP Compute images that eliminate slow dependency installation at sandbox start. Use when the base setup would take more than 30 seconds or the same environment is shared across multiple tracks.

## What It Provides

A custom GCP Compute image referenced by the `image:` field in `config.yml`. Packer builds the image from a base OS image (e.g., `ubuntu-os-cloud/ubuntu-2204-lts`), runs provisioners to install packages, pull Docker images, compile binaries, and configure the OS. The resulting image is stored in a GCP project and referenced by tracks at launch time.

## Config.yml Example

```yaml
version: "3"
virtualmachines:
  - name: workstation
    image: my-gcp-project/my-custom-image       # custom image built with Packer
    machine_type: n1-standard-4
    shell: /bin/bash
    nested_virtualization: true
```

### Packer Workflow (Overview)

1. Write a Packer template targeting a GCP base image.
2. Run `packer build` to create the image in your GCP project.
3. Reference the image as `<gcp-project>/<image-name>` in `config.yml`.
4. Use setup scripts for track-specific files (source code, configs) -- keep those out of the image.

## Common Pitfalls

- **Slow author iteration loop**. Every image change requires a Packer rebuild (5--15 minutes). Start with setup scripts during development; move to a custom image once the dependency stack stabilizes. See `decision-frameworks/custom-image-vs-heredoc.md` for the full decision framework.
- **Stale images**. Dependencies baked into images drift over time (expired GPG keys, renamed packages, deprecated APIs). Schedule periodic rebuilds or pin versions explicitly.
- **Baking track-specific content into the image**. Project files, config files, and anything the learner will see or edit should be generated by setup scripts so they stay in the track repository and are easy to update.
- **Image not accessible to Instruqt**. The GCP project hosting the image must grant read access to the Instruqt sandbox service account. A missing IAM binding causes sandbox launch failures with no obvious error.
- **Oversized images**. Large images increase sandbox start time despite avoiding install steps. Clean up apt caches, tmp files, and build artifacts before the image snapshot.
