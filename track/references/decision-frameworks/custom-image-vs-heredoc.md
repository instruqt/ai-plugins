# Custom Image vs Heredoc/Setup Script

Deciding whether to bake dependencies into a custom VM/container image (Packer) or generate project files inline via heredocs and setup scripts.

## Decision Criteria

### Use a custom image when:

- Dependency installation takes more than 30 seconds (large packages, Docker image pulls, compiled binaries).
- The dependency stack is large or fragile (specific compiler versions, OS-level patches, kernel modules).
- The same base environment is reused across multiple tracks.
- Learner experience requires fast sandbox startup (every second of setup time is friction).
- Dependencies are unlikely to change between track iterations.

### Use heredoc/setup-script generation when:

- The project is small (fewer than 10 files).
- Project content changes often during iteration and authoring.
- The track is a one-off or prototype.
- The generated files are the teaching content itself (learners should see them appear).
- Setup time is acceptable (under 30 seconds).

## Trade-offs

| Factor | Custom Image | Heredoc/Setup Script |
|--------|-------------|---------------------|
| Learner startup time | Fast -- dependencies pre-baked | Slower -- installs happen at sandbox start |
| Author iteration speed | Slow -- requires Packer rebuild on changes | Fast -- edit scripts and re-test |
| Self-contained track repo | No -- image lives in a registry | Yes -- everything in the track repo |
| Reproducibility | High -- image is immutable | Depends on external package versions |
| Maintenance burden | Must rebuild when dependencies update | Scripts may break on upstream changes |
| Track portability | Tied to image registry access | Fully portable |
| Disk / registry cost | Images consume storage | No additional storage |
| Debugging | Harder -- must inspect image layers | Easier -- read the setup script |

## Hybrid Approach

Most production tracks use both: a custom image for heavy base dependencies (Docker, k3s, language runtimes) and setup scripts for track-specific configuration (project files, config files, service setup).

```
Custom image (Packer):          Setup script (heredoc):
  - Docker CE                     - docker-compose.yml
  - Node.js 22                    - application source files
  - PostgreSQL 16                 - .env configuration
  - Pre-pulled Docker images      - systemd service units
```

## Recommendation

**Start with setup scripts during authoring.** The fast iteration cycle (edit script, test track) is more valuable during development than a few seconds of learner startup time.

**Move to a custom image before publishing.** Once the track is stable, bake slow-installing dependencies into a custom image to improve the learner experience.

**Keep project files in setup scripts even with custom images.** Files that represent teaching content (source code, configs the learner will edit) should be generated by setup scripts so they are visible in the track repository and easy to update.
